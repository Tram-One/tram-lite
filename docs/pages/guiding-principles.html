Guiding Principles

The following are principles that motivate the design philosophy around the interface and methods found in the Tram-Lite library.

1. Provide an HTML Experience First

While Javascript is a very powerful tool, the current interfaces around Web-Component definitions requires familiarity with concepts like classes, state management, lifecycles events, and more. Libraries like React have considerably raised the technical floor for would-be developers in the Frontend space.

We believe that a native HTML interface will enable more developers to enter the frontend space, and should allow developers to progressively learn Frontend technologies and concepts, without first requiring a strong javascript foundation.

2. Favor No-Build Environments

Tram-Lite's target environment is a basic html file. It should offer everything needed, and let developers build immediately, with no other tools or libraries required. While many frameworks will recommend generators or platforms for an optimal development experience, we should strive to make the HTML and vanilla javascript experience ideal, where only a static web host may be available.

3. Defer to Native APIs

Tram-Lite aspires to use and work with native web APIs as much as possible, including Web-Components, native element attributes, and custom event. New functionality should always preference these interfaces, over coming up with novel solutions.

4. Not a Framework

Tram-Lite is a library, it is not a framework. As much as possible, users should run and orchestrate their own code. In the face of bloat and tradeoffs associated with frameworks, and the difficulty associated with learning new syntax and data models, we'd like Tram-Lite to be a simple alternative. With respect to the Frameworkless Manifesto, we believe that Tram-Lite can be an alternative that offers few tradeoffs, and seamless integration with other libraries or frameworks.
